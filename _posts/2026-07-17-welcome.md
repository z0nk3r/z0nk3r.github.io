This is the first post under the new design. Going forward this space is for security research, CTF writeups, and technical notes — the usual mix of "I broke this" and "here's how". Yes I like the style of the site. Cringe? Sure. Mine? Hell yeah.

## What's here

The new theme leans on a monochrome cream/near-black color scheme, a distinct display font for titles, real syntax highlighting in code blocks, and a table of contents that follows you down the page.

A few things worth knowing before you start reading (or writing):

- **Tags and categories** — shown as small badges under the title, each linking to a `/tags/` page listing everything with that tag
- **Read time** — optional, set it in the front matter or leave it out
- **Cover images** — optional `image:` front matter field; posts without one, like this one, just show a placeholder instead

If you want to see how a link looks inline, [here's one](https://z0nk3r.github.io) — same weight as the surrounding text, just underlined.

## Posts and References

This site actually has two kinds of content. **Posts** — like this one — are dated, chronological, and live under [Blog](/blog/). **References** are the opposite: undated, evergreen lookup material meant to be found later rather than read start to finish, living under [References](/references/). The [HTTP status codes](/references/http-status-codes/) page is a real example already up — no narrative, just tables. Both share the same tag system, so a single tag page can mix posts and references together.

## Formatting reference

Everything below doubles as a working reference for writing new content — copy whatever's useful.

### Code blocks

Here's a small Python snippet, mostly so the syntax highlighting has something to chew on:

```python
def is_prime(n):
    if n < 2:
        return False
    for i in range(2, int(n ** 0.5) + 1):
        if n % i == 0:
            return False
    return True

primes = [n for n in range(2, 50) if is_prime(n)]
print(primes)
```

> Code blocks get a small header showing the detected language, and the colors are borrowed from the Everforest palette — sage green for strings, tan for keywords and types, muted gray for comments.

Not every block needs a language — a plain fenced block still gets the same header, just without a language label:

```
_posts/
└── 2026-07-17-welcome.md
```

Either way there's a Copy button in the corner.

### Tables

Tables can set their own column widths right in the markdown, instead of being stuck with the sitewide default split. Here's the front matter this post itself uses:

| Field | Required | Notes |
|---|---|---|
| `layout` | Yes | Always `post` |
| `title` | Yes | Post title |
| `date` | Yes | Publish date |
| `categories` | No | Shown as badges |
| `tags` | No | Shown as badges |
| `excerpt` | No | Falls back to a truncated version of the content |
| `read_time` | No | Integer minutes, omit to hide |
| `image` | No | Cover image path, omit to skip the image block entirely |
| `author` | No | Overrides the site-wide default author |

### Text formatting

Regular markdown all works as expected: **bold**, _italic_, `inline code`, and [links](/references/) render inline. Ordered lists work too:

1. Pick a date and a title
2. Drop a file in `_posts/` or `_references/`
3. Add front matter
4. Write

Nothing fancier than that.

## Table of contents

The sidebar on this page is generated from every `h2`/`h3`/`h4` in the article — it's not hand-written.

### How headings map to the tree

Headings nest by level: an `h3` attaches under the `h2` above it, an `h4` under the nearest `h3`, and so on. Skip a level — an `h2` followed directly by an `h4` — and it just attaches one level up instead of erroring.

#### Wrapping long headings that don't fit the sidebar on one line

This heading is deliberately long enough to wrap onto a second line in the sidebar. When that happens, the tree's vertical connectors still carry through the wrapped line instead of stopping after the first one — it's rendered as real monospace text with the wrap points computed ahead of time, not CSS trying to fake it after the fact.

## Publishing a new post

New posts go in `_posts/` as `YYYY-MM-DD-title.md`. New references go in `_references/` and use `layout: reference` instead of `layout: post` — no date needed, no TOC, no prev/next nav, just the content.

1. Create the file with the right front matter
2. Add any new tags to `_tag_pages/` if they don't already exist
3. Run `bundle exec jekyll serve` and check it locally
4. Commit and push
